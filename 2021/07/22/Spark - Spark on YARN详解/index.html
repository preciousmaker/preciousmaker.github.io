Spark - Spark on YARN详解

发表于2021-07-22|更新于2021-07-22|Spark
|阅读量:

Spark on YARN

https://www.cnblogs.com/bigdata1024/p/12116621.html

其实不仅仅是Spark，MR Flink HBase等都可以跑在YARN上
关键就在于Application Master(AM)

–master yarn
启动后产生四个进程
YarnCoarseGrainedExecutorBacked
YarnCoarseGrainedExecutorBacked
ExecutorLauncher
SparkSubmit

YARN有重试机制，具体参数如下
spark.yarn.maxAppAttempts yarn.resourcemanager.am.max-attempts in YARN 默认重试2次

Client模式

–deploy-mode client 默认就是client，也可以设置为 cluster
AM：负责申请资源
Driver：与AM通信，负责调度

因为Driver运行在本地，jars、作业的jar包、spark_conf内的东西，打成三个包，上传到HDFS，运行日志到这部分会卡顿

Cluster模式

如果是在cluster模式，Driver是运行在AM中，既申请资源，又负责调度

例如运行官方给的计算PI的例子，client模式下，控制台相当于一个Driver，可以看到输出结果
但是cluster模式下，看不出详细的日志和输出结果，需要去 yarn logs –application 查看
也可以用 yarn application -kill 停止任务

spark-shell 不能跑在cluster上，会报错，因为spark-shell启动后要进行交互

资源分配策略

当在YARN上运行Spark作业，每个Spark executor作为一个YARN容器运行。Spark可以使得多个Tasks在同一个容器里面运行。

具体可以参考之前的文章：YARN - 调优

对于client模式：nodemanager管理资源>=executor个数executor资源(内存+cpu)
对于cluster模式：nodemanager管理资源>=executor个数executor(内存+cpu）+driver资源（内存+cpu）

Data Locality

数据本地性 UI上也可以看到

Data locality is how close data is to the code processing it.
数据本地性是数据与处理它的代码的接近程度。

分为以下几类，性能越来越差
PROCESS_LOCAL 数据与运行的代码是在同一个进程中，例如内存cache的数据和运行的代码都在一起。这是最佳的本地性
NODE_LOCAL 数据与运行的代码在同一个节点上
NO_PREF 从任何地方同样快速地访问数据，并且没有位置偏好，从别的地方或者本地都可以
RACK_LOACL 数据与运行的代码在同一个服务器机架上。数据位于同一机架上的不同服务器上，因此需要通过网络发送，通常是通过单个交换机
ANY 数据在网络上的其他地方，而不是在同一个机架中

没有数据，只能降级，如果没有CPU了，可以等到CPU空闲再去运行，不降级，有等待时间(spark.locality.*)
