简介

MapReduce存在的一些局限性

  1. 从编程角度来看，不方便
  2. 一般来讲 map ==> 磁盘，reduce ==> HDFS；但如果一个作业由多个MR作业构成
    例如 : A ==> B ==> C ==> D，A的输出交给B，B的输出交给C等等，就会涉及磁盘IO，网络IO以及序列化等问题
  3. Map Task和Reduce Task都是进程级别，需要启动JVM，先忽略JVM复用，启动和销毁有一定的开销
  4. MR不能合理充分的使用内存
  5. map的结果是会自动排序，耗时
  6. 只能处理离线批处理，不能处理实时数据

批处理、离线处理：MR Hive Pig
流式、实时处理：Storm SS SSS Flink

流批一体：Spark Flink
可以以集群的方式部署
也可以跑在YARN上

只需要下载一个Spark就可以使用
Spark SQL
Spark Streaming
MLlib(机器学习)
GraphX(图形计算)
这些底层都是Spark Core

速度

官方介绍为快如闪电的统一分析引擎
内存计算快于磁盘计算，并且Spark启动是基于线程
有DAG，A ==> B ==> C ==> D 就不用每一步都落地，将各个之间打通，形成pipeline
不过不管是Spark还是MR，shuffle都是落地到磁盘的

易用

相同的需求，spark的代码量和MR不是一个数量级
并且支持Java, Scala, Python, R, and SQL.
还提供了80+个高阶函数(算子)

运行环境

可以运行在Hadoop(运行在YARN，读取HDFS数据), Apache Mesos, Kubernetes, standalone, 或者是cloud。
并且可以访问(读写)各种数据源。

y : Minor version(add APIs/features)

能选择最新的尽可能选最新的
